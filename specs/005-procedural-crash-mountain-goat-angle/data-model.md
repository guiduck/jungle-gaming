# Data Model: Procedural Crash Mountain and Goat Angle

## Scope

This feature has a frontend-only visual data model. It does not add backend entities, persistence
columns, API fields, RabbitMQ events, or stored frontend preferences.

## Existing Inputs

The visual projection consumes existing state from `frontend/src/stores/game-store.ts`:

- `phase: RoundStatus`
- `displayedMultiplierBps: number`
- optionally the current `round` snapshot for phase/scene context already available in the store

`displayedMultiplierBps` is a server-derived projection in integer basis points. The frontend uses
it for rendering only; it does not determine cashout, payout, crash, wallet balance, or history.

## Frontend Projection Types

Recommended helper types:

```ts
export interface CurvePoint {
  xPercent: number;
  yPercent: number;
}

export interface CrashMountainOptions {
  visualMaxMultiplier: number;
  xStartPercent: number;
  xEndPercent: number;
  yStartPercent: number;
  yEndPercent: number;
  sampleCount: number;
  minAngleDeg: number;
  maxAngleDeg: number;
  frameWidth: number;
  frameHeight: number;
}

export interface CrashMountainProjection extends CurvePoint {
  angleDeg: number;
  pathPoints: CurvePoint[];
  progressIndex: number;
}
```

The final implementation may adjust names, but it should keep these concepts explicit and
unit-testable.

## Default Visual Constants

Initial defaults should be local frontend constants:

- `visualMaxMultiplier`: `3`
- `xStartPercent`: lower-left/base position with enough goat padding
- `xEndPercent`: upper/right summit position with enough goat padding
- `yStartPercent`: lower mountain/base position
- `yEndPercent`: upper mountain/summit position
- `sampleCount`: enough points for a smooth SVG path without overcomplication, for example 32
- `minAngleDeg`: approximately `-8`
- `maxAngleDeg`: approximately `42`
- `frameWidth` / `frameHeight`: visual frame ratio used for tangent angle calculation

These constants are not product configuration. They should not be loaded from backend APIs or
persisted per user.

## Projection Rules

Multiplier normalization:

- Treat missing, non-finite, zero, negative, or below-`10000` bps values as `1.00x`.
- Convert basis points to multiplier with `multiplierBps / 10000`.
- Clamp the visual multiplier to `[1, visualMaxMultiplier]`.

Position:

- `xPercent` is based on logarithmic progress:
  `ln(multiplier) / ln(visualMaxMultiplier)`.
- `yPercent` is based on multiplier height:
  `(multiplier - 1) / (visualMaxMultiplier - 1)`.
- Both coordinates are clamped to configured bounds.
- Coordinates must be monotonic forward/upward for increasing valid multipliers.

Angle:

- Angle is derived from the same curve used for position.
- The implementation may use the analytical derivative from the spec or sampled neighboring
  points.
- The final value must be clamped to the configured readable angle range.

Path samples:

- `pathPoints` are generated by sampling the same curve from `1.00x` through
  `visualMaxMultiplier`.
- Samples must remain within scene bounds.
- The visible progressed segment uses these same samples and reveals right-to-left as required by
  ADR-008.

## State and Lifecycle Behavior

- Betting or settled phase: goat uses the idle sprite near the base projection.
- Running phase: goat uses the run sprite and follows `displayedMultiplierBps`.
- Crashed phase: goat uses the jump sprite and remains visually related to the last projected
  multiplier point.
- Reconnect/refetch jumps: projection may snap or ease to the new server-derived multiplier, but it
  must not invent intermediate payout truth.
- Above-ceiling multiplier: goat position and angle clamp at the visual summit while multiplier text
  continues to display the current `displayedMultiplierBps`.

## Non-Model Changes

No changes are planned to:

- `Round`, `Bet`, `Wallet`, `Money`, or `CrashPoint` domain models.
- PostgreSQL schemas or MikroORM mappings.
- shared package contract types.
- REST, WebSocket, or RabbitMQ payloads.
- Docker Compose environment variables.
